# Web UI

Status: `proposed`

## Problem

Glitch needs a usable first-pass UI that can surface monitor state, projects, agents, historical process data, and live process output without forcing the user to bounce between multiple tools or deep navigation stacks.

The UI also needs a development workflow that does not require rebuilding and re-embedding a static bundle on every frontend change.

## Goals

- build the UI as a SolidJS SPA under `src/web`
- use Vite for local development and production bundling
- keep the browser-side UI talking only to monitor-hosted API routes
- support embedded static serving later through the monitor binary
- support live process stream viewing with a bounded circular buffer
- establish a small internal component library and design system
- define a first-pass monitor API contract that the UI can build against
- align the visual language with the Marathon-inspired references:
  - flat
  - hard-edged
  - dense
  - technical
  - signage and print influenced
  - light-theme leaning with sharp contrast

## Non-Goals

- full UI implementation of logger search in v1
- polished animations or high-motion interaction design
- a broad generic component framework
- dark theme parity in the first pass
- complex routing or screen-to-screen app-shell chrome

## Visual Direction

The UI should feel:

- synthetic
- data-heavy
- rectangular
- structured like a dashboard, console, or shipping label
- low-chrome and high-information

The reference set suggests:

- monospaced or mono-adjacent typography
- crisp boxes and separators
- strong alignment and modular grid layout
- restrained but vivid accent colors
- small control surfaces and compact spacing
- labels, codes, markers, and metadata presented at the same hierarchy as primary content

The UI should avoid:

- rounded modern SaaS styling
- oversized empty spacing
- soft shadows and blurred glass
- deep drill-down navigation for data that can be surfaced side-by-side

## Monitor API Code Layout

Monitor-owned resources should live under the monitor feature and expose feature-local API code.

Recommended layout:

- `src/monitor/app/monitor/api/router.ts`
- `src/monitor/app/monitor/api/api-shapes.ts`
- `src/monitor/app/monitor/api/event-shapes.ts`
- `src/monitor/app/monitor/api/handlers/get-monitor.ts`
- `src/monitor/app/monitor/api/handlers/get-project.ts`
- `src/monitor/app/monitor/api/handlers/get-project-list.ts`
- `src/monitor/app/monitor/api/handlers/get-agent.ts`
- `src/monitor/app/monitor/api/handlers/get-agent-list.ts`
- `src/monitor/app/monitor/api/handlers/get-process.ts`
- `src/monitor/app/monitor/api/handlers/get-process-list.ts`

Rules:

- `router.ts` exports `createMonitorRouter(deps)`
- Bun server setup calls that router factory and registers the returned routes
- dependencies should be passed directly into the router factory for now
- use one handler file per endpoint action
- use singular action names
- handlers should convert request params and query into typed input objects, then produce typed output objects

## Handler Pattern

Each endpoint handler should follow the same structure:

1. create the input record from request params and query
2. validate it through its data class or schema
3. run the feature action
4. create an output record
5. write the output record as the HTTP response

The handler should not pass raw request state deeper into the app when a typed input object is available.

## HTTP Helpers

Shared HTTP helpers should exist so basic response writing does not get duplicated.

Recommended shared utilities:

- `okJson`
- `notFoundJson`
- `badRequestJson`
- `serverErrorJson`

These helpers should live in shared code and be reused across monitor and agent APIs.

## Routing

The UI should use Solid Router.

Recommended package:

- `@solidjs/router`

The first pass should still use explicit route registration, but the file and folder layout should mirror route paths so route ownership stays obvious.

## Vite and Build Strategy

The project should use a custom Vite root so the web app can coexist with the agent and monitor code in the same repository.

Planned Vite setup:

- Vite root: `src/web`
- entry HTML: `src/web/index.html`
- entry script: `src/web/main.ts`
- production build output: a dedicated web bundle directory such as `dist/web`
- Solid plugin enabled for Vite
- router-aware SPA fallback in development and production serving

The monitor will later embed the built static assets into the binary. For this first phase, the build should still produce normal static assets that can be consumed by the monitor build pipeline later.

## Dev Workflow

The frontend must support a local dev-server workflow.

Recommended approach:

1. run the monitor normally for APIs
2. run the Vite dev server separately for the web app
3. configure the Vite dev server to proxy monitor API and stream endpoints
4. open the Vite URL in the browser during UI development

This keeps the UI code unchanged:

- the browser still uses monitor-shaped endpoints such as `/api/*` and `/stream/*`
- in development, Vite proxies those routes to the running monitor
- in production, the monitor serves the bundled assets itself and the same paths resolve locally

Recommended dev routes:

- Vite origin serves HTML, JS, CSS, and HMR
- Vite proxies:
  - `/api/*`
  - `/stream/*`

Recommended dev environment shape:

- monitor runs on its own localhost port
- Vite runs on its own localhost port
- web code talks to relative paths only
- Vite proxy rewrites those relative paths to the running monitor

Recommended monitor behavior for this phase:

- no special monitor-side dev proxy logic is required
- the monitor continues to expose APIs as normal
- the browser uses the Vite dev server only while developing the UI

## Monitor API Surface

The first UI pass needs enough monitor-hosted API to browse system state and agent history.

The monitor should define explicit HTTP and WebSocket transport shapes using the same pattern used elsewhere in the repo:

- Zod schemas for runtime validation
- data classes for parsed monitor-side and browser-side payloads
- separate internal monitor shapes and public HTTP or Socket.IO shapes when the transport contract differs

All monitor API payloads should use `snake_case`.

The monitor owns these resources logically, even when some of the underlying data is read from registry rows, live agent APIs, live agent events, or per-project databases.

The monitor should keep both live and historical process inspection inside the same feature surface.

Rules:

- do not split historical process browsing into a separate monitor-side feature yet
- process inspection belongs to monitor
- live stream forwarding and historical database reads are internal implementation details behind the same monitor-owned resource model
- the UI should experience these as one coherent surface instead of separate live and historical subsystems

### Monitor

- `GET /api/monitor`
  - monitor status
  - version
  - start_date
  - pid
  - base_url

Suggested response:

```json
{
  "pid": 12345,
  "start_date": "2026-03-02T18:00:00.000Z",
  "base_url": "http://127.0.0.1:19001",
  "version": "0.1.0",
  "healthy": true
}
```

Suggested classes:

- `GetMonitorInput`
- `GetMonitorOutput`

### Projects

- `GET /api/projects`
  - list known projects
- `GET /api/projects/:project_id`
  - project details
  - latest agent summary

Suggested list item:

```json
  "cwd": "C:/Users/juanka/dev/me/glitch2",

  "last_ping_date": "2026-03-02T18:12:00.000Z",

```

Suggested classes:

- `GetProjectListInput`
- `GetProjectListOutput`
- `GetProjectInput`
- `GetProjectOutput`

### Agents

### Historical Process Data

The `process_runs` data is stored in the per-project agent database, not the registry.

The monitor should resolve the project cwd from the registry and then read:

- `<cwd>/.glitch/agent.glitch`

The monitor should open project agent databases in readonly mode only.

Requirements:

- the monitor never writes to a project agent database
- the monitor may keep readonly connections cached for reuse
- connection reuse should be keyed by project or cwd
- stale cached connections should be closed when no longer needed or when the source file is unavailable

Initial routes:

- `GET /api/agents/:agent_id/processes`
  - list `process_runs` for that agent
- `GET /api/agents/:agent_id/processes/:process_run_id`
  - process details

Suggested process item:

```json
{
  "id": "uuid",
  "agent_run_id": "uuid",
  "process_id": "web",
  "command": "bun run dev",
  "cwd": "C:/Users/juanka/dev/me/glitch2",
  "pid": 40120,
  "start_date": "2026-03-02T18:10:03.000Z",
  "end_date": null,
  "exit_code": null,
  "signal": null,
  "status": "running"
}
```

Suggested classes:

- `GetProcessListInput`
- `GetProcessListOutput`
- `GetProcessInput`
- `GetProcessOutput`

### Live Stream API

The monitor should provide a live stream endpoint for active process output.

Transport decision:

- use WebSockets
- use Socket.IO as the high-level transport library
- use the Bun Socket.IO engine integration
- expose one shared app-level endpoint:
  - `/api/event`

Recommended namespace or channel shape:

- Socket.IO client connects to `/api/event`

Suggested events:

- `stream.snapshot`
- `stream.line`
- `stream.clear`

Suggested client messages:

- `subscribe`
- `unsubscribe`

Suggested subscription message:

```json
{
  "agent_id": "uuid",
  "process_id": "web"
}
```

Suggested subscription classes:

- `SubscribeInput`
- `UnsubscribeInput`

Suggested line payload:

```json
{
  "agent_id": "uuid",
  "process_id": "web",
  "process_run_id": "uuid",
  "capture_date": "2026-03-02T18:12:04.000Z",
  "stream": "stdout",
  "line": "listening on http://localhost:3000"
}
```

Suggested event wrapper classes:

- `StreamSnapshotEvent`
- `StreamLineEvent`
- `StreamClearEvent`

For the first pass, the monitor should keep an in-memory circular buffer per active process stream.

Buffer rules:

- default keep last `100` messages
- buffer size should be configurable
- store normalized line entries, not raw chunks
- when the UI connects, replay the current buffer first, then stream live updates

## Event Bus

The Socket.IO integration should be wrapped in a shared `EventBus`.

Responsibilities:

- accept a Socket.IO server instance
- manage connect lifecycle
- handle generic `subscribe` and `unsubscribe` messages
- join and leave rooms
- expose high-level `emit` and `emitTo` methods

Recommended API:

- `emit(eventName, payload)`
- `emitTo(roomName, eventName, payload)`

The goal is to decouple app features from direct Socket.IO mechanics.

## Room Naming

Room names should follow topic-like naming.

Examples:

- `agent:<agent_id>`
- `agent:<agent_id>:process:<process_id>`

Do not construct room names inline throughout the codebase.

Provide helper functions such as:

- `getAgentRoomName(agentId)`
- `getProcessStreamRoomName(agentId, processId)`

## UI Information Architecture

The first pass should avoid deep routing and instead favor high-context layouts.

Recommended top-level views:

- Overview
  - monitor status
  - active agents
  - recent projects
  - recent failures
- Project
  - project summary
  - known agents for the project
  - latest process runs
- Agent
  - agent metadata
  - process list
  - live or historical process output

The agent view should keep:

- agent metadata
- process selector
- process stream view
- process run details

visible within the same page when possible.

The first pass should favor split panes, side-by-side summaries, and visible adjacent context over drill-down only navigation.

The UI should assume the monitor is the only data source.

The monitor is responsible for deciding whether data comes from:

- the registry database
- a live agent HTTP API
- a live agent event stream
- a readonly per-project agent database

The UI should not manage those distinctions itself.

Build a small internal component set instead of pulling in a large UI framework.

First-pass component set:

- button
- icon button
- panel
- data table or data grid
- split layout
- toolbar
- badge
- status pill
- field row
- log stream viewport
- empty state
- loading block

Each component should live in its own folder:

- `src/web/components/button/*`
- `src/web/components/panel/*`
- `src/web/components/log-stream/*`

Recommended contents:

- component TSX
- component CSS
- component-local helper files if needed
- component-local transport or view-model shapes if truly local

## CSS Rules

Use the `gl-` prefix for all shared classes.

Examples:

- `gl-button`
- `gl-panel`
- `gl-data-grid`
- `gl-log-stream`

Modifier style should stay compact.

Examples:

- `.gl-button.size-large`
- `.gl-panel.tone-critical`
- `.gl-log-stream.compact`

Prefer modern scoped CSS where practical so component-local styling stays compact and understandable.

Recommended rule:

- use a root `gl-*` class per component
- use scoped styles when supported by the current toolchain and browser target
- keep global utility styling minimal
- use Tailwind only as a small augmentation layer, not as the primary component styling system
- colocate CSS with the component or route it styles
- keep `src/web/styles/*` limited to global tokens, resets, and app-wide styles that do not belong to a single component

Established components should be styled with dedicated colocated CSS files and design tokens rather than inline utility accumulation.

## Design System

The design system should be small and explicit.

Define:

- spacing scale
- padding scale
- border widths
- font stack
- font sizes
- line heights
- neutral palette
- accent palette
- semantic states

Recommended tokens:

- `--gl-space-*`
- `--gl-font-*`
- `--gl-color-*`
- `--gl-border-*`
- `--gl-radius-*`
- `--gl-size-*`
- `--gl-z-*`

The initial look should bias toward:

- near-white or light gray surfaces
- dark text
- one strong green or cyan accent
- one warning orange
- one error red
- subtle grays for grids and separators

Recommended spacing behavior:

- compact default vertical rhythm
- narrow panel padding
- tight but readable table rows
- explicit tokenized gaps instead of ad hoc margins

## State Management

Keep state management simple in v1.

Recommended approach:

- Solid signals and stores for page state
- small resource wrappers for monitor APIs
- a dedicated live-stream store per active stream view
- parsed data classes created from validated HTTP and Socket.IO payloads before the UI consumes them

Avoid introducing a large client state framework unless the monitor API complexity proves it necessary.

## Stream View Behavior

The live stream panel is the most important first-pass feature.

Requirements:

- show stdout and stderr lines as they arrive
- color-code stream source
- keep stream output dense and readable
- support pause or freeze later, but not required in phase 1
- preserve only the last N lines in memory
- allow switching between active processes quickly

Suggested line fields:

- capture_date
- process_id
- stream
- line

## Browser Client Wrappers

The browser should use typed wrappers instead of raw fetch and raw socket wiring.

Recommended wrappers:

- `MonitorApiClient`
- `MonitorEventSource`

Rules:

- keep API client code separate from event-source code
- validate incoming payloads before passing them into view code
- do not expose raw socket message payloads directly to components
- the UI does not need an `AgentEventSource` because it never talks to agents directly

## Accessibility and Usability

Even with the dense aesthetic, the UI should remain readable.

Requirements:

- keyboard-accessible navigation
- visible focus states
- strong text/background contrast
- compact but not cramped line height
- no color-only meaning for critical status

## Implementation Phases

### Phase 1

- set up Vite with `src/web` as the root
- configure Solid plugin and Solid Router
- add Solid entrypoint and base app shell
- define design tokens and base styles
- define HTTP and Socket.IO transport shapes with Zod-backed data classes
- implement monitor, project, and agent list/detail APIs in the monitor
- implement feature-local API router and handler layout under `src/monitor/app/monitor/api/*`
- implement shared response helpers for JSON endpoints
- implement shared `EventBus` with Socket.IO room subscriptions
- implement readonly project-agent database access and connection reuse in the monitor
- implement overview, project, and agent routes
- implement live process stream view with Socket.IO and circular buffer

### Phase 2

- expand historical browsing for process runs
- add richer process detail panels
- add better filtering and sorting
- add UI polish for dense dashboards

### Phase 3

- embed bundled assets into the monitor binary
- remove the need for external static file hosting in production
- add richer logger search and cross-view correlations

## Testing Scenarios

- Vite dev server resolves `src/web/index.html` and `src/web/main.ts`
- Vite dev proxy forwards monitor API routes correctly
- Vite dev proxy forwards monitor Socket.IO routes correctly
- monitor list routes return stable validated JSON payloads
- project and agent detail routes resolve correctly
- monitor can read historical `process_runs` from a project database through readonly connections
- live stream buffers replay the last N messages on connection
- buffer truncation preserves only the configured window
- UI renders monitor, project, and agent states without layout collapse on dense data
- component-local CSS remains colocated and does not leak unexpectedly into unrelated views

## Open Questions

- which exact monospaced font stack should define the final visual identity
- how much of the UI bundle embedding work should live in the monitor build versus a shared build script

## Notes for Future Revisions

The first UI pass should bias toward useful information density over navigation ceremony.

The right measure of success is that a developer can open Glitch and immediately answer:

- what projects are known
- what agents are active
- what processes are running
- what output is happening now
- what happened recently

without bouncing through a maze of screens.
