# Logger Module

Status: `proposed`

## Problem

Local development produces a constant stream of useful information, but it is fragmented across terminals, browser consoles, framework overlays, and log files.

Glitch needs a logging foundation that can:

- capture their output reliably
- preserve raw fidelity
- make logs searchable and filterable
- stream updates live to the UI
- retain historical runs for later inspection

## Goals

- Capture both `stdout` and `stderr` from supervised processes
- Persist logs into a per-project SQLite database
- Support both live viewing and historical querying
- Preserve the raw emitted line while also leaving room for structured parsing later
- Associate logs with agent runs and process runs
- Support full-text search in v1 through SQLite FTS
- Keep the initial design simple enough to implement incrementally

## Non-Goals

- Cross-machine log aggregation
- Production deployment support
- Rich log parsing for every framework format in v1
- Metrics, request tracing, or event modeling in the logger module

## Configuration

The initial design assumes a root project config file:

`glitch.config.json`

The logger-specific portion of the config looks like this:

```json
{
  "name": "my-app",
  "modules": {
    "logger": {
      "enabled": true,
      "retention_days": 14
    }
  }
}
```

Process definitions are owned by the supervision feature and documented in [`20260228-supervision.md`](./20260228-supervision.md).

## Responsibilities

The agent is responsible for process supervision.

The logger module is responsible for:

- subscribing to process lifecycle events emitted by the agent
- subscribing to stdout/stderr stream events from supervised processes
- buffering stream data until a full line is available
- writing line-oriented log records into SQLite
- exposing log data to the UI
- publishing live log updates through the agent transport layer

## Storage Model

Each project stores data locally at:

`<projectRoot>/.glitch/glitch.sqlite`

The logger stores one row per completed log line.

Each stored row should capture:

- the line as emitted by the process
- the process and run it came from
- whether it came from `stdout` or `stderr`
- the timestamp when Glitch captured it
- an `event` JSON column for future structured parsing

The `event` column should be nullable in v1. The logger should not attempt to fully parse framework-specific formats yet.

The logger depends on the supervision feature for `agent_runs` and `process_runs`.

## SQLite Schema

### `log_entries`

Stores line-oriented log records.

Columns:

- `id` uuid (text)
- `process_run_id` uuid (text) references `process_runs.id`
- `seq` integer (integer)
- `stream` process_stream (text)
- `capture_date` datetime (text)
- `line` string (text)
- `source?` string (text)
- `event?` json (text)

### Indexes

Recommended initial indexes:

- `log_entries(process_run_id, seq)`
- `log_entries(capture_date)`
- `log_entries(stream, capture_date)`

### Full-Text Search

FTS is required in v1.

Use SQLite FTS to index log content cheaply without introducing extra infrastructure.

Recommended approach:

- create an FTS virtual table for `log_entries`
- index `line`
- optionally index `source` later if it proves useful
- keep the base `log_entries` table as the source of truth

## Ingestion Pipeline

1. Agent starts and loads `glitch.config.json`
2. Agent ensures `.glitch/` and the SQLite schema exist
3. Supervision creates `agent_runs` and `process_runs`
4. Logger subscribes to process start, process stop, stdout, and stderr events from the agent
5. As stream data arrives:
   - append incoming bytes/text to a per-process, per-stream buffer
   - wait until a newline is encountered
   - emit exactly one log event per completed line
   - persist exactly one `log_entries` row per completed line
6. On process exit:
   - flush any remaining buffered content as a final line if content exists
7. On agent shutdown:
   - stop accepting new stream data

## Parsing Rules

- `log_entries` are the source of truth for logger persistence in v1
- the logger buffers stream data internally and only persists complete lines
- the emitted line should be preserved in `line`
- the `event` column remains `null` in v1 unless a very safe parser is introduced later
- parsing strategy beyond raw capture is deferred
- if ANSI escape sequences are present, preserve the original line in storage and handle display cleanup separately in the UI if needed

## UI Access

The UI should access logger data in two ways:

- live: via agent HTTP + WebSocket
- historical: via the project SQLite database

Suggested API endpoints:

- `GET /health`
- `GET /project`
- `GET /runs/current`
- `GET /logs?process_id=&cursor=&limit=`
- `GET /logs/search?query=&process_id=&limit=`
- `WS /stream`

Suggested live events:

- `run.start`
- `run.stop`
- `process.start`
- `process.exit`
- `log.write`

## Failure Modes

The design should explicitly handle:

- abrupt process exit
- agent crash during capture
- incomplete final lines
- very noisy output streams
- multiple configured processes
- stale UI connections
- database initialization on first run

## Testing Scenarios

- captures stdout and stderr separately
- preserves stream order per process
- persists one row per complete emitted line
- flushes incomplete trailing line on process exit
- associates each log line to the correct `process_run_id`
- supports paginated log queries
- supports FTS queries over stored log lines
- emits one live update per completed log line
- survives invalid or high-volume log lines without corrupting storage

## Phased Implementation

### Phase 1

- define config types
- create logger-specific SQLite schema
- implement logger subscriptions to agent lifecycle and stream events
- persist line-oriented log entries
- add SQLite FTS support

### Phase 2

- add agent HTTP and WebSocket endpoints
- support live UI streaming
- add basic filtering and pagination

### Phase 3

- add retention and pruning behavior
- improve level detection
- evaluate full-text search

## Notes for Future Revisions

The logger is intended to be the foundation for later modules.

Future modules such as requests, events, and metrics may reference `agent_runs` and `process_runs` to correlate all monitoring data under a shared execution model.
